# План развития S-Art

## 1. Цель

Развить текущий локальный прототип S-Art до безопасной, устойчивой и
масштабируемой социальной платформы для публикации визуальных работ и хранения
промптов.

План предусматривает постепенное развитие без преждевременного усложнения:

1. стабилизировать существующую функциональность;
2. закрыть основные риски безопасности;
3. подготовить приложение к эксплуатации;
4. устранить ограничения локального файлового хранилища;
5. внедрить горизонтальное масштабирование;
6. развивать социальные и творческие функции.

## 2. Текущее состояние

### Архитектура

- Vue 3 SPA в Nginx-контейнере;
- FastAPI REST API с несколькими Uvicorn workers;
- две PostgreSQL-базы для демонстрационного и рабочего режимов;
- Redis-кеш ленты, промптов и статистики;
- локальный Docker volume для загруженных изображений;
- Docker Compose и healthcheck сервисов;
- JWT-авторизация;
- два backend worker по умолчанию;
- демо-данные и изображения из локального каталога и Pexels;
- роли `user` и `admin`.

### Основные ограничения

- локальные файлы привязывают данные к одному серверу;
- production-секреты пока требуют ручной настройки через `.env`;
- отсутствует полноценный механизм миграций;
- нет refresh-токенов, отзыва сессий и восстановления пароля;
- проверяется расширение файла, но не его фактическое содержимое;
- отсутствуют rate limiting, аудит действий и защита от автоматических атак;
- нет автоматических тестов и CI/CD;
- нет мониторинга, централизованных журналов и оповещений;
- список публикаций кешируется, но пока возвращается без пагинации;
- отсутствуют фоновые задачи;
- удаление записи не всегда означает удаление связанного файла;
- внешние изображения могут стать недоступны.

## 3. Целевые уровни нагрузки

Масштабирование следует выполнять на основании измерений, а не предполагаемой
нагрузки.

| Уровень | Пользователи | Одновременно онлайн | Запросы | Хранилище |
|---|---:|---:|---:|---:|
| Прототип | до 1 000 | до 50 | до 10 RPS | до 20 ГБ |
| Малый production | до 20 000 | до 500 | до 100 RPS | до 500 ГБ |
| Средняя платформа | до 200 000 | до 5 000 | до 1 000 RPS | до 10 ТБ |
| Крупная платформа | более 1 млн | более 20 000 | более 5 000 RPS | десятки ТБ |

Начальная целевая SLO для малого production:

- доступность API — 99,9% в месяц;
- p95 чтения ленты — не более 300 мс;
- p95 операций записи — не более 600 мс;
- доля ответов 5xx — менее 0,5%;
- успешная загрузка изображений — не менее 99%;
- RPO резервного копирования — 24 часа;
- RTO восстановления — 4 часа.

После достижения среднего масштаба:

- доступность — 99,95%;
- p95 чтения — до 200 мс;
- RPO — 15 минут;
- RTO — 1 час.

## 4. Приоритеты

Обозначения:

- `P0` — обязательно до публикации в интернете;
- `P1` — обязательно для стабильного production;
- `P2` — необходимо при росте;
- `P3` — перспективное развитие.

## 7. Этап 2 — защита приложения

Срок: 3–5 недель. Приоритет: `P0`.

### Управление секретами

- `JWT_SECRET`, URL базы и параметры хранилища получать из окружения;
- использовать уникальный криптографически стойкий секрет;
- хранить production-секреты в Secret Manager/Vault;
- определить процедуру ротации;
- запретить вывод секретов и токенов в журналы.

### Авторизация и сессии

- короткоживущий access-токен;
- refresh-токен с ротацией;
- хранение refresh-токена в `HttpOnly`, `Secure`, `SameSite` cookie;
- таблица активных сессий;
- отзыв отдельной или всех сессий;
- выход со всех устройств;
- подтверждение email;
- восстановление пароля через одноразовую ссылку;
- ограничение попыток входа;
- возможность MFA для администратора;
- повторное подтверждение пароля перед критическими операциями.

### Пароли

- минимальная длина не менее 10–12 символов;
- проверка распространённых и скомпрометированных паролей;
- сохранение bcrypt или переход на Argon2id;
- отсутствие ограничений, снижающих надёжность длинных паролей;
- нейтральный ответ при восстановлении пароля, не раскрывающий существование
  email.

### Защита API

- rate limiting по IP, пользователю и маршруту;
- отдельные лимиты для входа, регистрации, загрузки и запросов в друзья;
- ограничение размера HTTP-запроса на уровне reverse proxy;
- строгая валидация длины всех строк;
- серверный перечень допустимых категорий;
- запрет массового присваивания полей;
- корректные CORS-настройки только для известных доменов;
- security headers: CSP, HSTS, X-Content-Type-Options, Referrer-Policy;
- защита от IDOR с обязательной проверкой владельца;
- запрет доверия к переданным клиентом `user_id` и `role`;
- единый безопасный обработчик исключений.

### Файлы и изображения

- проверять MIME-сигнатуру, а не только расширение;
- декодировать изображение безопасной библиотекой;
- удалять EXIF и потенциально чувствительные метаданные;
- ограничивать разрешение и число пикселей;
- защищаться от decompression bomb;
- перекодировать изображения в безопасный формат;
- генерировать миниатюры;
- проверять файлы антивирусом;
- запретить SVG от пользователей либо выполнять строгую очистку;
- выдавать файлы с корректным `Content-Type`;
- использовать случайные непрогнозируемые имена;
- удалять осиротевшие файлы.

### Администрирование

- MFA для администраторов;
- журнал всех действий администратора;
- причина удаления контента;
- мягкое удаление с возможностью восстановления;
- запрет удаления последнего администратора;
- разделение ролей: модератор, администратор, супер-администратор;
- повторное подтверждение критических действий.

### Критерии завершения

- устранены критические и высокие замечания OWASP;
- выполнен dependency audit;
- проведён ручной security review;
- критические действия попадают в неизменяемый журнал аудита.

## 8. Этап 3 — production-инфраструктура

Срок: 3–5 недель. Приоритет: `P1`.

### Контейнеризация

- отдельные Dockerfile для frontend и backend;
- multi-stage сборка frontend;
- запуск backend от непривилегированного пользователя;
- read-only root filesystem, где это возможно;
- healthcheck и readiness check;
- лимиты CPU и памяти;
- Docker Compose для локальной интеграционной среды.

### Reverse proxy

- Nginx, Traefik или облачный load balancer;
- HTTPS;
- HTTP/2 или HTTP/3;
- сжатие текстовых ресурсов;
- кеширование статических файлов;
- ограничения тела запроса;
- таймауты соединений;
- корректная передача client IP.

### CI/CD

Pipeline pull request:

1. установка зависимостей;
2. линтеры и типизация;
3. unit- и интеграционные тесты;
4. frontend build;
5. проверка зависимостей;
6. сканирование секретов;
7. сборка контейнера;
8. сканирование образа.

Pipeline release:

1. создание версионированных образов;
2. развёртывание staging;
3. миграции;
4. smoke-тесты;
5. ручное или автоматическое подтверждение;
6. production deployment;
7. автоматический rollback при сбое.

### Среды

- development;
- test;
- staging;
- production.

Данные, секреты и внешние сервисы сред не должны пересекаться.

## 9. Этап 4 — PostgreSQL и управление данными

Срок: 3–6 недель. Приоритет: `P1`.

### Переход с SQLite

- внедрить Alembic;
- создать начальную версионированную схему;
- перейти на PostgreSQL;
- подготовить скрипт миграции существующих данных;
- проверить внешние ключи и каскадные правила;
- добавить уникальные ограничения;
- добавить индексы;
- выполнить тестовую миграцию и rollback.

### Предлагаемые индексы

- `users(email)` — уникальный;
- `posts(created_at DESC)`;
- `posts(user_id, created_at DESC)`;
- `posts(category, created_at DESC)`;
- `prompts(user_id, created_at DESC)`;
- `favorites(user_id, item_type, item_id)` — уникальный;
- `friendships(user_id, friend_id)` — уникальность нормализованной пары;
- `friendships(friend_id, status)`;
- `sessions(user_id, expires_at)`;
- полнотекстовые индексы для поиска.

### Целостность данных

- запрет дублирующихся дружеских связей;
- запрет повторного добавления одного объекта в избранное;
- внешние ключи для всех владельцев;
- транзакции для сложных операций;
- единая стратегия временных зон — UTC;
- поля `created_at`, `updated_at`, `deleted_at`;
- soft delete для пользователей и контента;
- периодическая очистка окончательно удалённых данных.

### Подключения

- пул соединений;
- таймаут запросов;
- обнаружение медленных запросов;
- read replica при достижении соответствующей нагрузки.

## 10. Этап 5 — объектное хранилище и обработка медиа

Срок: 3–6 недель. Приоритет: `P1`.

### Хранение

- S3-совместимое хранилище: AWS S3, MinIO или облачный аналог;
- отдельные buckets для оригиналов и производных файлов;
- versioning и lifecycle policy;
- серверное шифрование;
- запрет публичной записи;
- подписанные URL при необходимости;
- хранение в БД ключа объекта, а не абсолютного домена.

### Конвейер изображения

```text
Клиент
  → временная загрузка
  → проверка файла
  → антивирус
  → удаление метаданных
  → перекодирование
  → миниатюры
  → объектное хранилище
  → CDN
```

### Варианты размеров

- thumbnail: 320 px;
- card: 720 px;
- large: 1600 px;
- оригинал — только при необходимости.

Использовать WebP/AVIF с fallback. Не передавать полноразмерный оригинал в
ленте.

### CDN

- длительное кеширование по версионированному URL;
- `Cache-Control: public, max-age=31536000, immutable`;
- географическое распределение;
- защита origin от прямого массового скачивания;
- purge при модерации или удалении.

## 11. Этап 6 — производительность API

Срок: 3–6 недель. Приоритет: `P1`.

### Пагинация

Все списки перевести на cursor pagination:

- лента;
- публикации пользователя;
- промпты;
- друзья;
- входящие запросы;
- избранное;
- административные списки.

Ответ должен содержать `items`, `next_cursor` и признак наличия следующей
страницы. Не использовать неограниченную выдачу всех записей.

### Оптимизация запросов

- устранить N+1 запросы при получении авторов;
- использовать SQLAlchemy eager loading;
- выбирать только необходимые поля;
- агрегировать счётчики без загрузки всех объектов;
- анализировать `EXPLAIN ANALYZE`;
- задать таймауты;
- ограничить максимальный размер страницы.

### Кеширование

Redis использовать для:

- rate limiting;
- короткого кеша общей статистики;
- горячих страниц ленты;
- статусов сессий;
- одноразовых токенов;
- блокировок фоновых задач.

Не кешировать персональные данные без чёткой стратегии ключей и инвалидации.

### Сжатие и протокол

- Brotli/Gzip для JSON, JS и CSS;
- HTTP/2;
- ETag или Last-Modified;
- условные запросы;
- минимизация JSON-ответов.

## 13. Этап 8 — горизонтальное масштабирование

Срок: по результатам нагрузки. Приоритет: `P2`.

### Stateless backend

Backend не должен хранить локальное пользовательское состояние. Все экземпляры
используют:

- общую PostgreSQL;
- Redis;
- объектное хранилище;
- общий сервис секретов.

После этого можно запускать несколько экземпляров за балансировщиком.

### Масштабирование

- минимум два backend-экземпляра в production;
- автоматическое масштабирование по CPU, памяти, latency и RPS;
- отдельное масштабирование HTTP API и workers;
- graceful shutdown;
- readiness probe, исключающая неготовый экземпляр из балансировки;
- rolling update без простоя;
- Pod Disruption Budget при Kubernetes.

### База

Порядок масштабирования PostgreSQL:

1. оптимизация запросов и индексов;
2. увеличение ресурсов;
3. connection pooler, например PgBouncer;
4. read replicas;
5. разделение тяжёлой аналитики;
6. партиционирование крупных таблиц;
7. шардирование только при доказанной необходимости.

Не вводить микросервисы и шардирование до появления измеримого узкого места.

- тестировать staging, максимально похожий на production;
- использовать синтетические данные;
- не направлять нагрузочные тесты на production без отдельного разрешения;
- фиксировать версию кода, инфраструктуры и набор данных;
- сравнивать результаты между релизами;
- задавать performance budget.

### Начальный критерий

При 100 RPS и 500 одновременных пользователях:

- p95 чтения менее 300 мс;
- p95 записи менее 600 мс;
- ошибки менее 1%;
- отсутствует устойчивый рост памяти;
- очередь фоновых задач возвращается к норме после пика.

## 16. Этап 11 — резервное копирование и аварийное восстановление

Срок: 2–3 недели. Приоритет: `P0/P1`.

### Резервные копии

- ежедневный полный backup PostgreSQL;
- непрерывное архивирование WAL для point-in-time recovery;
- versioning объектного хранилища;
- копия в другом регионе или аккаунте;
- шифрование;
- документированный срок хранения;
- автоматическая проверка успешности.

### Восстановление

- ежеквартальное учебное восстановление;
- проверка целостности базы и файлов;
- runbook для дежурного;
- фиксация фактических RPO и RTO;
- сценарий утечки ключа;
- сценарий повреждения базы;
- сценарий удаления bucket;
- сценарий недоступности региона.

Backup считается рабочим только после успешной проверки восстановления.

## 21. Целевая архитектура

```text
Пользователь
    │
    ▼
CDN / WAF
    │
    ├── статический Vue frontend
    │
    ▼
Load Balancer / Reverse Proxy
    │
    ├── FastAPI instance 1
    ├── FastAPI instance 2
    └── FastAPI instance N
            │
            ├── PostgreSQL + replica
            ├── Redis
            ├── S3-compatible storage
            └── Queue → background workers

Все компоненты → logs / metrics / traces / alerts
```

На малом масштабе компоненты могут работать в Docker Compose. Kubernetes
оправдан только при наличии команды, нескольких сервисов и необходимости
автоматического масштабирования.

## 22. Предлагаемые изменения API

- версионирование: `/api/v1`;
- единый формат ошибки;
- cursor pagination;
- idempotency key для операций создания;
- correlation ID;
- OpenAPI-схема как контракт;
- ограничение размера страниц;
- optimistic concurrency через версию или `updated_at`;
- отдельный endpoint загрузки медиа;
- статус обработки изображения;
- bulk endpoints только с ограниченным размером;
- документированная политика совместимости.

Пример ошибки:

```json
{
  "error": {
    "code": "POST_NOT_FOUND",
    "message": "Работа не найдена",
    "request_id": "01H...",
    "details": {}
  }
}
```

## 23. Предлагаемая модель ролей

| Роль | Возможности |
|---|---|
| Пользователь | Собственный профиль и контент |
| Модератор | Жалобы, скрытие контента, ограничения пользователей |
| Администратор | Пользователи, роли, контент и настройки |

Разрешения желательно хранить отдельно от названия роли и проверять на backend.
Frontend только скрывает недоступные элементы, но не является средством защиты.

## 27. Реестр основных рисков

| Риск | Вероятность | Влияние | Мера |
|---|---|---|---|
| Утечка JWT-секрета | Высокая | Критическое | Переменные окружения, ротация |
| Вредоносная загрузка | Средняя | Критическое | Декодирование, AV, MIME, лимиты |
| Потеря базы или файлов | Средняя | Критическое | Backup и тест восстановления |
| Перегрузка SQLite | Высокая при росте | Высокое | PostgreSQL |
| Недоступность локальных файлов при масштабировании | Высокая | Высокое | S3 и CDN |
| Массовый подбор паролей | Высокая | Высокое | Rate limit, MFA, мониторинг |
| Удаление контента администратором по ошибке | Средняя | Высокое | Soft delete, audit, подтверждение |
| N+1 и выдача всей ленты | Высокая | Среднее/высокое | Eager loading, пагинация |
| Зависимость от внешних URL изображений | Средняя | Среднее | Импорт в собственное хранилище |
| Рост стоимости инфраструктуры | Средняя | Высокое | Метрики, budgets, lifecycle |
| Спам и нежелательный контент | Высокая | Высокое | Жалобы, модерация, лимиты |
| Нарушение авторских прав | Средняя | Высокое | Лицензии, правила, takedown |
