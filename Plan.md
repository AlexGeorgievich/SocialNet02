# План развития S-Art SocialNet02

## 1. Исходная точка

Вторая версия уже прошла базовую архитектурную переработку. Следующие компоненты
не являются будущими задачами — они реализованы и проверены:

- Docker Compose;
- Vue 3 и Nginx;
- FastAPI с двумя Uvicorn workers;
- PostgreSQL 16;
- отдельные базы `sart_demo` и `sart_work`;
- Redis 7 с AOF и политикой `allkeys-lru`;
- кеширование ленты, промптов и статистики;
- инвалидация кеша после изменений;
- устранение N+1-запроса авторов в ленте;
- healthcheck всех контейнеров;
- постоянные Docker volumes;
- автоматическое демо-наполнение;
- изоляция JWT между demo/work;
- UTF-8-логи;
- production-сборка frontend;
- Vite 8 и отсутствие известных npm-уязвимостей;
- документация Docker-запуска.

Контрольный тест кешированной ленты показал 216–243 RPS без ошибок против
примерно 23 RPS у первой SQLite-версии.

## 2. Цель

Развить текущую Docker-версию из демонстрационного приложения в безопасную,
наблюдаемую и горизонтально масштабируемую production-платформу.

Приоритеты:

- `P0` — необходимо до публичного запуска;
- `P1` — необходимо для стабильной эксплуатации;
- `P2` — необходимо при росте аудитории;
- `P3` — продуктовое и долгосрочное развитие.

## 3. Текущая архитектура

```text
Пользователь
    │
    ▼
Nginx
    ├── Vue SPA
    ├── API proxy
    └── uploads proxy
           │
           ▼
FastAPI / Uvicorn workers
    ├── sart_demo PostgreSQL
    ├── sart_work PostgreSQL
    ├── Redis cache
    └── uploads Docker volume
```

### Достигнутая производительность

| Endpoint | Условия | Результат |
|---|---|---:|
| Кешированная лента | concurrency 10 | 216 RPS |
| Кешированная лента | concurrency 30 | 243 RPS |
| p95 ленты | concurrency 30 | 153 мс |
| Ошибки | 2 × 500 запросов | 0 |

Это локальный короткий тест. Для production-планирования необходимы k6/Locust,
длительные серии и данные, сопоставимые с реальным объёмом.

## 4. Оставшиеся архитектурные ограничения

- схема создаётся через SQLAlchemy, но миграции не версионируются;
- изображения находятся в локальном Docker volume;
- отсутствует пагинация;
- инвалидация очищает весь кеш вместо точечных ключей;
- отсутствуют refresh-токены и отзыв сессий;
- нет rate limiting;
- нет фоновой очереди;
- нет автоматических тестов и CI/CD;
- нет централизованных метрик и трассировки;
- отсутствует системная модерация;
- внешние Pexels URL могут стать недоступны;
- PostgreSQL работает в одном экземпляре;
- резервное копирование не автоматизировано;
- Docker Compose рассчитан на один хост.

## 23. Реестр рисков

| Риск | Влияние | Мера |
|---|---|---|
| Дефолтные секреты | Критическое | Обязательный production `.env`/Secret Manager |
| Один PostgreSQL | Высокое | Backup, managed DB, replica |
| Локальный uploads volume | Высокое | S3/MinIO |
| Нет Alembic | Высокое | Версионированные миграции |
| Вредоносный файл | Критическое | MIME, decode, AV, resize |
| Cache stampede | Среднее | Lock, jitter, stale-while-revalidate |
| Полный сброс кеша | Среднее | Namespace versioning |
| Публичные demo-пароли | Среднее | Не включать demo в production |
| Нет rate limit | Высокое | Redis limiter |
| Нет проверенного backup | Критическое | Автоматизация и restore drill |
| Внешние Pexels URL | Среднее | Импорт в собственное хранилище |
| Спам и контент | Высокое | Модерация и ограничения |

План пересматривается после каждого нагрузочного теста, инцидента, крупного
релиза или изменения требований законодательства.
